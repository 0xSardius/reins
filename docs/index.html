Why this exists

Agents are buying their own inputs now. Raw keypairs don't say no.

Data, inference, search, RPC — the pay-per-request rails already move real volume on Solana. But an agent holding a bare keypair has no concept of enough. reins is the missing middle layer: a wallet that says yes fast enough for micropayments and no firmly enough to run overnight.

max-per-payment

Per-service caps

No single call to api.x.com may cost more than the cap you set — '0.10' means ten cents, ever.

daily-budget

Rolling budgets

Per-service and global 24-hour budgets, plus a lifetime cap. Only settled payments count.

cooldown

Cooldowns

A minimum gap between payments to the same service. Denials tell the agent exactly how long to wait.

allowlist

Allow & block lists

Pay only the services you named — by origin or hostname. Blocklist always wins.

approval

Human in the loop

Payments over a threshold (or to flagged services) wait for your hook: Slack ping, push, CLI prompt.

ledger

Durable ledger

Every payment lands in an append-only JSONL ledger, so budgets survive restarts and audits are trivial.

How a payment happens

One fetch. Five checkpoints. Zero unbounded spending.

  1. 402

    The API challenges

    Your agent calls a paid endpoint. The server answers 402 Payment Required — an x402 v1 body, an x402 v2 PAYMENT-REQUIRED header, or an MPP WWW-Authenticate: Payment challenge. reins recognizes all three.

  2. POLICY

    The policy engine rules

    The challenge becomes a normalized payment intent — service, amount, asset, network — and runs the full gauntlet: blocklist → allowlist → network → asset → caps → cooldown → rate limit → budgets → approval. A denial throws PolicyViolationError with the exact rule that fired.

  3. SIGN

    The transfer is signed

    reins builds the spec-exact SVM transaction — compute budget, TransferChecked USDC to the recipient's token account, memo — and signs with the agent key. On x402, the facilitator co-signs as fee payer but can never move funds: its address appears in no instruction.

  4. RETRY

    The request retries, paid

    The same request goes back with X-PAYMENT (v1), PAYMENT-SIGNATURE (v2), or Authorization: Payment (MPP) attached. The server verifies, settles, and serves the resource.

  5. RECORD

    The ledger remembers

    Settlement receipts and transaction signatures land in the ledger, feeding tomorrow's budget math and npx reins status.

Two protocols, one wallet

Speaks x402 and MPP. Your agent doesn't have to care.

x402

The volume rail

The HTTP 402 protocol that's settled tens of millions of payments on Solana. reins implements the exact SVM scheme natively.

  • v1 and v2 wire formats, auto-detected
  • Facilitator fee-payer co-signing (your agent needs no SOL)
  • Validated against the production x402.org facilitator
MPP

The HTTP-native standard

mppx ships Tempo and Stripe methods — reins contributes the Solana one: a solana/charge method any MPP server can verify.

  • On-chain USDC transfer, memo-bound to the challenge (no replay)
  • Server-side verifier included — monetize your own API
  • Composes with the whole mppx ecosystem
The whole integration

Policy is code you can read at 2am.

import { createReinsWallet } from 'reins'

const wallet = await createReinsWallet({
  policy: {
    dailyBudget: '5.00',            // ≤ $5/day across everything
    defaults: { maxPerPayment: '0.10', cooldownMs: 2_000 },
    allowlist: ['api.helpful-service.com'],
    approvalThreshold: '0.50',      // bigger spends ask a human first
    onApprovalRequired: (intent) => askSlack(intent),
  },
})

// 402s on this fetch are now paid automatically — within policy.
const res = await wallet.fetch('https://api.helpful-service.com/v1/answer')
